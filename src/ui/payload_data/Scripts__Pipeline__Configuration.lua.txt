[description]
Pipeline configuration script - coordinates asset compilation settings, imports GlobalSettings.lua and others, and handles command-line overrides for the Frostbite build pipeline.

[version]
2011 (Battlefield 3 Alpha, Command & Conquer: Generals 2)
[notes]
Original pipeline configuration with Maya integration.

Loading Order:
1. GlobalSettings.lua (common settings)
2. commandLine.startupScript (if specified - Drone custom settings)
3. MayaExport/Startup.lua (Maya DCC integration)
4. Settings.lua (game-specific settings)
5. commandLine.config (if specified - Drone custom settings)

Maya Version Detection:
- MAYA_VERSION environment variable
- Supports: 8.5 (850) and 2009

Command-Line Switches (Boolean):
- debug → Pipeline.DebugMode
- validateData → Pipeline.ValidateData
- addToPerforce → Pipeline.AddToPerforce
- noconsole → Pipeline.NoConsole
- verbose → Pipeline.Verbose
- trace → Pipeline.EnableTracing
- nohal → Pipeline.DisableHalDevice
- trim → Pipeline.TrimLayout
- updateIndex → Pipeline.UpdateIndexOnly

Command-Line Arguments (String):
- opfilter → Pipeline.OpFilter

Diagnostics:
- Pipeline.Verbose triggers settings dump and command-line dump

This represents the earliest asset pipeline configuration with Maya exporter integration.
[content]
--
-- Pipeline Configuration
--
--   This is an internal script used for configuring the pipeline. Normal
--   users should typically not edit this file, as most end-user settings
--   are exposed through the GlobalSettings.lua file
--

--
-- Import common settings
--

core.import(allScriptsRoot .. "GlobalSettings.lua", "global settings");

-- Import session startup script, if specified on commandLine
-- 
-- This is used by Drone to change settings

if commandLine.startupscript then
	core.import("/data/" .. commandLine.startupScript, _G)
end

--
-- Import Maya exporter settings
--

core.import(allScriptsRoot .. "MayaExport/Startup.lua", "MayaExport configuration")

core.import(scriptsRoot .. "Settings.lua")

--   -config commandLine switch is used by Drone to configure custom settings
if commandLine.config then
	core.readKeyValueCfg("/data/Scripts/" .. commandLine.config, _G)	
end

--------------------------------------------------------------------------
-- Configuration scripts will set the MAYA_VERSION environment variable
-- to indicate which Maya version to use for the current branch
--

local MayaVersion = os.getenv("MAYA_VERSION")

if MayaVersion == "8.5" then
	MayaDependencies.MayaVersion = 850
elseif MayaVersion == "2009" then
	MayaDependencies.MayaVersion = 2009
end

--------------------------------------------------------------------------
-- Handle commandline options
--
--   Commandline options effectively override any settings specified
--   elsewhere.
--

local function cmdLineSwitch(switchName, fieldName)
	if (commandLine[string.lower(switchName)]) then
		Pipeline[fieldName] = true
	end
end

local function cmdLineArg(switchName, fieldName)
	local entry = commandLine[string.lower(switchName)]
	
	if (entry) then
		Pipeline[fieldName] = entry
		--print("Set " .. fieldName .. " to " .. entry)
	end
end

cmdLineSwitch("debug", "DebugMode");
cmdLineSwitch("validateData", "ValidateData");
cmdLineSwitch("addToPerforce", "AddToPerforce");
cmdLineSwitch("noconsole", "NoConsole");
cmdLineSwitch("verbose", "Verbose");
cmdLineSwitch("trace", "EnableTracing");
cmdLineSwitch("nohal", "DisableHalDevice");
cmdLineSwitch("trim", "TrimLayout");
cmdLineSwitch("updateIndex", "UpdateIndexOnly");

cmdLineArg("opfilter", "OpFilter")

--
--  Parse commandline options that may occur multiple times ("list" options)
--

----------------------------------------------------------------------------------------
--
-- Diagnostics dump
--

if Pipeline.Verbose then
	print "
	print "Pipeline settings summary:"
	print "------------------------------------------------------"
	
	for k,v in pairs(Pipeline) do
		print(k, v)
	end
	
	print "

	print "Command line:"
	print "------------------------------------------------------"
	
	for _,v in pairs(commandLine) do
		print(_,v)
	end

	print "
end

--
-- Parse raw PipelineSettings and CacheSettings struct values
--
-- This ensures they can be found when the C++ SettingsManager asks the Lua
-- settings provider for them.
--
-- Custom pipeline settings objects return by createSettings() are not parsed
-- here, but instead directly from the command line on the C++ side
-- (PipelineApp::applySettingsOnContainer()), where we also account for
-- sub/super classes.

for index,option in ipairs(commandLine) do
	local table,member = string.match(option, "^%-(%w+)%.([%w%.]+)")

	if table and member and (table == "Pipeline" or table == "Cache") then
		local value = commandLine[index+1] 
		if not _G[table] then _G[table] = {} end
		_G[table][member] = value 
	end
end

2025 (Battlefield 6, NFL 26, Skate Playtest)
[notes]
ARCHITECTURAL REVOLUTION: Licensee separation + simplified loading.

This represents a fundamental reorganization of the pipeline configuration system - the most significant architectural change since the original 2011 implementation.

Major Architectural Changes:

1. **Licensee/Frostbite Separation**:
   - Frostbite owns: Settings.lua, Warnings.lua (core defaults)
   - Licensees own: Licensee/Settings.lua, Licensee/Warnings.lua (game-specific overrides)
   - Clear separation of concerns between engine and game teams

2. **New Loading Order**:
   1. commandLine.startupscript (if specified)
   2. Settings.lua (Frostbite defaults)
   3. Licensee/Settings.lua (game overrides) - OPTIONAL
   4. /data/Scripts/LocalPipeline.cfg (developer overrides)
   5. Warnings.lua (Frostbite warnings)
   6. Licensee/Warnings.lua (game warnings) - OPTIONAL
   7. commandLine.config (Drone custom settings)

3. **Optional Licensee Files**:
   - core.import() with {required=false} flag
   - Licensee/Settings.lua and Licensee/Warnings.lua are optional
   - Allows Frostbite-only configurations without game-specific files

4. **Removed Legacy Systems**:
   - Removed: GlobalSettings.lua (deprecated since 2011)
   - Removed: allScriptsRoot (unified to scriptsRoot)
   - Removed: MayaExport support (FBX workflow standard)
   - Removed: trace, nohal, functionParameters switches

5. **LocalPipeline.cfg**:
   - New developer override file: /data/Scripts/LocalPipeline.cfg
   - Loaded AFTER licensee settings
   - Allows per-developer customization without modifying version-controlled files

6. **Simplified Command-Line**:
   - Essential switches only: debug, validateData, addToPerforce, noconsole, verbose, trim, updateIndex
   - Removed obsolete switches from 2011-2024 era
   - opfilter argument removed

7. **ContentDatabase Support**:
   - Retained from 2021 version
   - Three struct types: ContentDatabase, Pipeline, Cache

Philosophical Shift:
- 2011-2024: Monolithic configuration with game-specific divergences
- 2025+: Layered configuration with clear ownership boundaries

This architecture enables:
- Clean Frostbite updates without breaking games
- Game teams can override without modifying core files
- Developer-specific settings without source control conflicts
- Future-proof extensibility

Comment Change:
- 'Normal users should typically not edit this file' → 'Licensee developers should not edit this file'
- Emphasizes Frostbite ownership vs game team customization

This is the modern template that will carry Frostbite pipeline configuration into the next generation.
